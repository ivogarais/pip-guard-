Technical Architecture Report

pip-guard Runtime Architecture

pip-guard defers package review to a dedicated Daytona sandbox. The sandbox stages the target release, resolves the dependency graph with uv, gathers evidence with security tools, asks Codex to decide allow / warn / block, and returns a structured verdict plus an HTML dossier before any local install is attempted.

sandbox: pip-package-security-checker model: gpt-5.4-mini transport: HTTP on Daytona preview URL install rule: exact scanned version only

1. System Summary

Execution boundary

Package staging, evidence generation, Codex review, and report rendering all occur inside Daytona. The local machine does not execute package installation until the sandbox has completed its review.

Decision boundary

Supporting tools gather facts. Codex is the policy and reasoning layer that converts those facts into a normalized verdict consumed by the wrapper.

Data boundary

Large outputs remain on the sandbox filesystem. The wrapper interacts with small JSON endpoints such as status, verdict, result, and report URLs.

2. Execution Flow

  1. The wrapper ensures the Daytona sandbox is started and the scan service is listening on port 3001.
  2. The wrapper submits a package string to POST /scan.
  3. The scan worker stages the package artifacts and resolves unpinned input to an exact version.
  4. uv resolves dependencies and emits a lock-backed view of the graph under review.
  5. The runner gathers advisory, code, secrets, SBOM, and heuristic evidence inside the sandbox.
  6. Codex reads the local evidence set and writes codex-verdict.json.
  7. The runner renders a markdown summary and HTML report, then the wrapper enforces the returned decision locally.
The wrapper handles orchestration and enforcement. The sandbox handles package analysis and verdict generation.

3. Core Components

Component Role Key behavior
Local wrapper Operator-facing entrypoint Starts the sandbox if needed, ensures the service is reachable, renders Rich progress, fetches verdict data, prompts on warn, denies on block, and installs the reviewed exact version.
Daytona scan service Job control API Exposes /scan plus per-job URLs, persists job state on disk, and launches the runner in a detached process.
Scan runner Evidence pipeline Stages package artifacts, resolves dependencies with uv, collects advisories, executes scanners, builds the evidence overview, and renders final report files.
Codex launcher Reasoning runtime Runs Codex inside the sandbox with persistent CODEX_HOME and a fast review model, currently gpt-5.4-mini.

4. Evidence Pipeline

The pipeline is intentionally staged. Early steps normalize the package under review, later steps add evidence, and the final step performs reasoning over the full evidence set.

staging
artifact fetch + unpack
resolution
uv lock + exact graph
advisories
PyPI / OSV matching
analysis
bandit, secrets, SBOM, heuristics
review
Codex verdict generation
report
summary + HTML dossier

Evidence types collected before Codex review

  • Resolution evidence: resolved dependency graph and exact package version.
  • Advisory evidence: vulnerability lookups associated with resolved packages.
  • Code evidence: Python-oriented static findings from bandit.
  • Secrets evidence: suspicious tokens and secret-like strings.
  • Inventory evidence: SBOM plus file inventory for artifact composition.
  • Heuristic evidence: pattern searches and file-type anomalies.

Codex receives the compact evidence overview plus local file paths, not a loose natural-language summary. That keeps the decision grounded in sandbox artifacts.

6. Decision Semantics

ALLOW

No material maliciousness or unacceptable risk is identified. The wrapper proceeds to install the exact scanned version automatically.

WARN

The package appears non-malicious but carries vulnerabilities or suspicious conditions that justify an operator stop. The wrapper prompts before installation.

BLOCK

The package is clearly malicious, deceptive, or dangerously abusive. The wrapper denies installation unless the operator explicitly overrides policy.

7. Operational Notes

Persistence

CODEX_HOME is persisted inside the sandbox runtime storage so Codex login survives reconnects without embedding credentials into an image.

Warm runtime

The checker sandbox is intended to remain warm, with the worker already available, so the wrapper does not pay a full cold-start penalty on each request.

Install safety

If the operator typed an unpinned package name, the install step still uses the resolved exact version that the sandbox reviewed, not the original loose input.
